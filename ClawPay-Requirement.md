# ClawPay - Agentic Settlement Protocol

## 1. 背景

从"对话 AI"到"行为 AI"的跃迁。

**趋势：** 市场正在进入 Machine Economy 时代。未来的互联网流量将不再仅由人类产生，大量的 API 调用、数据买卖和资源分配将由 Agent 独立决策完成。

**断层：** 虽然 Agent 已经可以写代码、订机票、做调研，但它们依然缺乏一个原生的经济身份。目前的金融体系（信用卡、银行转账）是为人类设计的，存在高延迟、高费率和复杂的 KYC 门槛，完全无法适配 Agent 毫秒级的交互需求。

## 2. 市场痛点分析

### 2.1 支付摩擦与"预付费"困境
- **现状：** 用户必须为每一个可能用到的 Agent 服务预存资金或绑定信用卡
- **痛点：** 这种模式极其低效，限制了 Agent 跨平台寻找最优服务的能力
- **需求：** 市场需要一种"按需支付、实时清算"的通用货币

### 2.2 机器间的"信任荒漠"
- **风险：** 预付可能导致 Provider 拿钱不干活；后付可能导致 Requester 拿到结果后赖账
- **需求：** 市场需要一个机器界的支付宝，通过代码（Smart Contract）来代替法律，建立机器间的信任

### 2.3 资产波动的干扰
- **影响：** 如果 Agent 之间用波动性资产结算，可能会出现任务执行到一半因汇率变动导致余额不足
- **USD1 的价值：** 作为合规且稳定的资产，USD1 提供了确定性的计价标准

## 4. 用户管理端：策略与身份中心 (The Cockpit)

这是 Human 掌握最高控制权的地方。

### A. 支出策略与 DID 绑定 (Policy & ID Editor)

| 功能 | 描述 |
|------|------|
| Root DID 状态 | 绑定 World ID 或 ENS，显示主信誉分（Global Reputation） |
| 子 Agent 管理 | 列表显示旗下所有 Agent（如：Research_Agent_01, Trading_Bot_X） |
| 单笔/日限额 | 基于 USD1 设置 |
| 信誉门槛 | 自动拒绝向信誉分低于 X 的 DID 发起支付 |
| 白名单 Mandate | 预授权特定的合约地址 |

### B. 资金面板 (The Financial Dashboard)

| 功能 | 描述 |
|------|------|
| 实时资金池 | 显示主钱包与各个 Agent 子钱包的 USD1 余额 |
| 任务完成度与 ROI 分析 | 自动抓取 Agent 支付的 USD1 换回的任务成果，计算单次任务成本 |
| 信誉溢价分析 | 统计因 Agent 性能不佳或对方信誉低导致的"额外支出" |

## 5. 身份隔离架构

| 层级 | 描述 | 链上表现 |
|------|------|----------|
| Human Root DID | 绑定 World ID / ENS | 主钱包地址，持有全部 USD1 |
| Agent Sub-DID | 由 Root 派生的唯一标识符 | 小额、受限的 MPC 钱包 |
| Mandate（授权令） | EIP-712 结构化签名 | 授权 Agent 在指定时间内最多支取 X USD1 |

**Mandate 示例：**
> "我（Human DID）授权我的 Agent A 在 24 小时内最多支取 50 USD1 用于支付给符合条件的 Provider。"

## 6. Agent 协议扩展 (The Protocol SDK)

### 核心接口

```
1. 身份握手 (DID Handshake)
   verify_identity(target_did)
   - 在支付前，Agent 自动查询对方是属于哪个"人"的
   - 是否存在历史黑名单记录

2. 自动对账 (Auto-Reconciliation)
   - Agent 每次支付后，自动向管理端推送加密的 Activity_Log
   - 包含：任务 ID、对方 DID、最终成交价（含动态系数）
```

```

├──────────────────────────────────────────────────────────────────────┤
│                                                                        │
│  1. Identity Match                                                     │

│     └→ 任务完成后释放资金给 Provider                                   │
│                                                                        │

```

## 8. Claw-Economics 动态定价模型

### 核心结算公式

```
Total_Cost = Base_Fee × K_Reputation × K_Complexity × K_Supply/Demand
```

### 系数定义

| 系数 | 描述 | 取值范围 |
|------|------|----------|
| K_Reputation | 基于 DID 关联信誉分实时调整 | 0.8 ~ 1.5 |
| K_Complexity | Provider 报价时提交的 Task_Manifest | 1.0 ~ 2.5+ |
| K_Supply/Demand | 监控全网任务队列动态调整 | 0.9 ~ 2.0 |

### K_Reputation 详细规则

| 信誉分 | 系数 | 说明 |
|--------|------|------|
| > 90 分 | 0.8x | 激励折扣 |
| 60-90 分 | 1.0x | 正常 |
| < 60 分 | 1.2x | 溢价惩罚 |
| < 40 分 | 1.5x | 信用破产，强制 50% 溢价进入保险池 |

### K_Complexity 详细规则

| 级别 | 系数 | 描述 |
|------|------|------|
| L1 (简单) | 1.0x | 单次 API 呼叫 |
| L2 (中等) | 1.5x | 多步推理/长文本 |
| L3 (复杂) | 2.5x+ | 跨 Agent 协作/高并发算力 |

### K_Supply/Demand 详细规则

| 状态 | 系数 | 描述 |
|------|------|------|
| 空闲 | 0.9x | 引导任务接入 |
| 正常 | 1.0x | 标准价格 |
| 峰值 | 1.2x ~ 2.0x | 类似滴滴动态加价 |

## 9. 信用体系评分

### 最终评分公式

```
Final_Score = (Human_Score × 0.7) + (Agent_Score × 0.3)
```

### Human Root Score (人：财务信用分)

- **初始分：** 75
- **核心：** 衡量该人类用户的全局违约风险与资金实力
- **计算来源：**
  - 资金沉淀：主钱包中 USD1 的平均余额与质押时长
  - 合规历史：旗下所有 Agent 产生纠纷的总频率
  - 实人认证：是否通过了 World ID 等高级别验证
- **变动逻辑：** 极其缓慢。建立高分需长期、多维度的合规表现

### Agent Sub-Score (机：业务表现分)

- **初始分：** 75
- **核心：** 衡量该特定 AI 任务逻辑的执行确定性与交付质量
- **计算来源：**
  - 任务成功率：单笔任务的交付回执通过率
  - 响应效率：任务接受到交付的平均时长
  - 复杂度溢价率：预估复杂度与实际消耗的偏差值
- **变动逻辑：** 反应灵敏。旨在通过短期的高频表现快速建立或损毁市场地位

## 10. 风险治理与保险机制 (The Safety Net)

利用低信誉 Agent 产生的 0.5x 惩罚性溢价构建系统屏障。

### 溢价回流 (Premium Feedback)

```
低分 Agent 支付的额外费用 → 锁定在托管合约
    │
    ├─ 任务成功 ─→ 80% 退还 Requester + 20% 注入协议保险池
    │
    └─ 任务违约 ─→ 溢价用于赔付 + 触发信誉断崖下跌
```

### 开源可配置性

提供 Economic Strategy SDK，允许企业在私有网络内 Override 预设系数（如内部 Agent 互通设为 0 费率）。

## 11. 商业模式：Agent 劳动力租赁 (Marketplace)

### 算力转售

- Human A 可以将高信誉、高性能的 Agent 出租给 Human B 使用

### 分润逻辑

- Agent 赚取的 USD1 实时结算至所属 Human A 的 Root 钱包
- ClawPay 仅抽取极小比例（如 0.1%）的协议费
